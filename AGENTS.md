# Инструкции для проекта LazaCup

## Язык
- Всегда отвечай на русском языке.
- Код, имена функций, переменных, классов, таблиц БД и путей не переводи.

## Общий режим работы
- Сначала анализируй задачу, потом предлагай план, потом вноси правки.
- Без явного подтверждения не меняй файлы.
- Без явного подтверждения не создавай новые файлы.
- Не трогай файлы, которые не относятся к задаче.
- Если задача неоднозначна, сначала перечисли варианты решения с плюсами и минусами.

## Работа с кодом
- Предпочитай минимальные точечные изменения.
- Не делай рефакторинг "заодно", если задача этого не требует.
- Не переименовывай существующие функции, переменные, таблицы и маршруты без необходимости.
- Сохраняй текущую архитектуру проекта и стиль кода.
- Перед изменением покажи, какие файлы будут затронуты и зачем.

## Работа с БД
- Сначала показывай SQL отдельно, потом предлагай его применение.
- Не выполняй destructive SQL без явного подтверждения.
- Если есть риск затронуть данные, сначала предложи безопасный способ проверки.
- Для проверки авторизации, расписания, публикации и архивации сначала опирайся на текущую схему таблиц и реальные связи в коде.

## Работа с локальной средой
- Проект работает локально в OpenServer.
- Для MySQL используй актуальные локальные настройки проекта, а не предположения.
- Не предлагай боевые Linux-пути как локальные пути.
- Если видишь жёсткие абсолютные пути, сначала укажи на них отдельно.

## Работа с задачами
- Для больших задач сначала:
  1. перечисли связанные файлы
  2. опиши текущую логику
  3. предложи план
  4. только потом переходи к правкам
- Для диагностики ошибок сначала найди точную причину, а не предлагай массовые изменения.

## Безопасность
- Не предлагай опасные команды без предупреждения.
- Не удаляй файлы, таблицы или данные без явного подтверждения.
- Если проблема может быть в окружении, путях, сессиях, cookie, БД или правах доступа, сначала проверь это, а потом предлагай изменения в бизнес-логике.

## Формат ответа
- Пиши кратко и по делу.
- Когда анализируешь, разделяй:
  - причина
  - что проверить
  - что менять
- Если уверен не полностью, прямо укажи степень уверенности.

## Контекст проекта
- Основные админские страницы находятся в /admin/organizer и /admin/operator.
- API проекта находится в /api/tournament.
- Перед изменением логики авторизации, расписания, публикации и архивации сначала найди связанные HTML, JS, PHP и SQL-части.
- Если задача затрагивает UI и API одновременно, сначала покажи полную цепочку файлов от страницы до API.
- Если задача касается логина, сессии или доступа, отдельно проверь cookie, session и ответ me.php.

## Ограничение широких задач
- Если задача слишком широкая, не обходи весь проект сразу.
- Сначала покажи промежуточный результат и предложи, как разбить задачу на этапы.

## Объём анализа
- Для первого шага предпочитай минимально достаточный набор файлов для MVP, а не полный аудит всего проекта.

## Порядок изменений
- Для каждой нетривиальной задачи сначала покажи:
  1. связанные файлы
  2. минимальный план
  3. что именно будет изменено
- Только после этого переходи к правкам.

## Проверка перед правками
- Перед предложением изменений сначала проверь, нет ли уже похожей логики в проекте.
- Не дублируй существующие helper-функции, проверки и API без необходимости.
